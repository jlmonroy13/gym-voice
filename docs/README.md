## Documentation map

This document is the entry point for GymVoice documentation. Start here to understand what to read first and how the main docs relate to each other.

- **PROJECT.md**: Repository operating contract, mission, scope boundaries, non‑negotiable principles, and how this repo is meant to be used.
- **docs/product/PRD.md**: Product Requirements Document — product goal, target users, MVP scope, primary flows, and success criteria.
- **docs/architecture/SOLUTION_DESIGN.md**: Technical solution design — architecture, layers, data flow, agents, skills, and how the system is structured to meet the PRD.
- **docs/architecture/STACK.md**: Stack and tooling — single source of truth for mobile/backend tech choices, testing tools, and infrastructure decisions.
- **docs/process/DEFINITION_OF_DONE.md**: Definition of Done — completion standard for issues and PRs, including stricter rules for voice pipeline, auth, and persistence work.
- **docs/process/TASK_EXECUTION_WORKFLOW.md**: Task execution workflow — how to pick issues, create branches, implement, validate, open PRs, and close work in alignment with the project board.

### What to read first

1. **PROJECT.md** — to understand the mission, scope, and non‑negotiables.
2. **docs/product/PRD.md** — for product context, MVP definition, and main user flows.
3. **docs/architecture/SOLUTION_DESIGN.md** and **docs/architecture/STACK.md** — for architecture, layers, and stack decisions.
4. **docs/process/TASK_EXECUTION_WORKFLOW.md** and **docs/process/DEFINITION_OF_DONE.md** — for how to execute tasks, validate work, and decide when something is actually done.
